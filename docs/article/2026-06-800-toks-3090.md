# 808.7 tok/s from one RTX 3090 Ti

808.7 tokens/sec of aggregate LLM throughput from a single RTX 3090 Ti. Eight concurrent streams, 4.40× scaling, ~101 tok/s per stream, coherent verified output. Getting there took four dead ends — and the fastest dead end was generating pure gibberish.

**The goal**

Parallel local agent workers on one 24 GB card. Single-stream was solved early: byteshape's Qwen3.6-35B-A3B under llama.cpp Vulkan runs 127.7 tok/s with 100% toolcall and 5/5 agentic. One excellent worker. We wanted eight.

**Dead end 1: a pool of slots is not a pool**

llama.cpp has slot parallelism. Four slots should land somewhere near 4× aggregate. Measured: 1.25× on Vulkan, 1.27× on CUDA. Four streams bought a quarter more tokens.

**Dead end 2: speculative decoding fights batching**

MTP looked great solo: +28% single-stream decode on Gemma 12B. At four streams on CUDA it inverts — 114.6 tok/s aggregate with spec, 174.8 without. −34% for leaving it on. Speculative decoding spends the compute headroom that batching needs. Turn it off past one stream.

**Dead end 3: 360 tok/s of nothing**

vLLM scaled. 2.76× at four streams, 360.3 tok/s aggregate, first model we tried. Then the coherence check: temperature 0, "say hello". The reply was multi-script token salad. Every prompt, gibberish.

We eliminated the tool parser, the chat template, and mrope config loss one by one. The same weights score 1.000 toolcall under llama.cpp. Remaining suspect: that VL-architecture community quant under vLLM 0.22.1.

The lesson: throughput suites can't see corruption. 360 tok/s of garbage measures exactly like 360 tok/s of code. Coherence-probe before you benchmark.

**Dead end 4: the quant format wall**

Next candidate died at startup: `Only symmetric quantization is supported for MoE`. Its compressed-tensors INT4 quant hits a MoE path vLLM 0.22.1 refuses. Classic AWQ/GPTQ formats avoid that path entirely.

**The unlock, in four takes**

Qwen3-30B-A3B, official GPTQ-Int4. 15.77 GiB of weights, text-only architecture. Still took four takes, each fixing exactly one thing:

1. Quant size. 35B quants don't fit 24 GB with batch KV cache; this one leaves room for 8 streams.
2. Quant format. Symmetric GPTQ — dodges the MoE assertion above.
3. Tool parser. Gen-3 Qwen speaks hermes JSON, not `qwen3_xml`. With the wrong parser the model emitted perfect tool calls the parser threw away as plain text.
4. Probe budget. A thinking model can't say hello in 128 tokens — the probe burned its whole budget in the reasoning channel and returned blank.

**The number**

| Streams | Aggregate tok/s | Scaling | Per-stream tok/s |
|---:|---:|---:|---:|
| 1 | 183.8 | — | 183.8 |
| 2 | 309.8 | 1.69× | 155.4 |
| 4 | 534.4 | 2.91× | 133.9 |
| 8 | 808.7 | 4.40× | 101.3 |

22,173 MiB loaded. Toolcall 0.972 over 36 trials. Coherent output, parsed tool calls, all verified.

The honest caveat: a 3-run agentic variance pass scored 3/5, 1/5, 2/5 — and the same two tasks failed every run. That's systematic, not noise. This config stays off autonomous multi-turn duty. Its lane is well-specified single-shot fan-out.

**Proof on real work**

32 real docstring items, 8 streams, 24.7 seconds, 32/32 AST-verified. On short-generation work the sustained number is 345 tok/s average with a 512 peak — prefill and verify gaps eat decode time. 808 is the long-decode ceiling. The full fan-out writeup is coming as a companion piece.

**Reproduce it**

vLLM 0.22.1 serving `Qwen/Qwen3-30B-A3B-GPTQ-Int4`:

```
--max-model-len 16384 --max-num-seqs 8 --gpu-memory-utilization 0.92 \
--quantization gptq_marlin --enable-auto-tool-choice \
--tool-call-parser hermes --reasoning-parser qwen3
```

Hardware: one RTX 3090 Ti, 24 GB. Harness, raw logs, model SHAs, and the full gibberish diagnosis: github.com/svankina/local_agents.

The card came out in 2022.
